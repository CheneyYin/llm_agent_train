贯穿演示的底层任务：分析 `sample.log` 的错误，定位 `parse.py` 的 bug。

---

### 步骤 1：无角色 vs 有角色

**前情过渡**：
"提示词工程听起来高大上，其实就是'学会怎么跟 AI 说话'。那我们从头开始——如果你只能做一件事来改进提示词，那就是给 AI 一个身份。"

**讲师话术**：
"我们先用最直接的方式问 AI——不附加任何条件。"

**命令（无角色）**：
```
分析 sample.log
```

**预期输出**：
- AI 给出泛泛的文本分析
- 可能不会主动提 parse.py 的代码问题
- 输出格式随意

**讲师话术**：
"你看，回答很泛。现在我们加一句话，给它一个身份。"

**命令（有角色）**：
```
你是一个资深 Python 后端开发，负责代码审查。请分析 sample.log
```

**预期输出**：
- AI 开始从开发者视角分析
- 提到日志级别分布、可能的代码问题
- 语言更专业

**关键展示点**：
- 同一任务，多加一句话，输出质量天差地别
- "资深 Python 后端开发"比"代码审查"这个身份更具体

**备用方案**：
- 若两轮输出差异不明显，追加角色描述："你是一个有 10 年经验的 Python 后端开发，专门处理日志系统"

**衔出过渡**：
"角色设定让 AI 知道了'我是谁'，但光有身份还不够——你还得告诉它'我要什么、不要什么'，就像去咖啡店只说"来杯咖啡"，不说清楚什么咖啡、多大杯、冰的热的——含糊不清一定拿不到你想要的。"

---

### 步骤 2：模糊指令 vs 精确指令

**前情过渡**：
"角色设定让 AI 知道了'我是谁'，但光有身份还不够——你还得告诉它'我要什么、不要什么'，就像去咖啡店只说"来杯咖啡"，不说清楚什么咖啡、多大杯、冰的热的——含糊不清一定拿不到你想要的。"

**讲师话术**：
"现在我们试试告诉 AI 具体要什么。先看模糊的。"

**命令（模糊指令）**：
```
你是一个资深 Python 后端开发。帮我分析这个日志文件。
```

**预期输出**：
- AI 输出自由文本，格式随意
- 可能包含一些无关信息

**讲师话术**：
"能看，但不够好。现在我们给精确的约束。"

**命令（精确指令）**：
```
你是一个资深 Python 后端开发。
分析 sample.log，要求：
1. 按错误类型分组（DatabaseError、InternalError 等）
2. 统计每种错误出现的次数
3. 按次数降序排列
4. 指出哪些错误可能是同一个根因导致的
```

**预期输出**：
- 结构清晰：分组、统计、排序都有
- 分析深度增加：开始推测根因

**关键展示点**：
- "要求"列表里的每一项都让 AI 输出更聚焦
- 告诉 AI 你要什么格式，它就会给你什么格式

**备用方案**：
- 若无明显差异，再加一条"请用中文回复"

**衔出过渡**：
"精确指令能让 AI 不跑偏，但有些东西——比如你想要的代码风格、分析深度——很难用一句话描述清楚。最直接的办法：给个参考答案。"

---

### 步骤 3：Few-shot（0-shot vs 2-shot）

**前情过渡**：
"精确指令能让 AI 不跑偏，但有些东西——比如你想要的代码风格、分析深度——很难用一句话描述清楚。最直接的办法：给个参考答案。"

**讲师话术**：
"现在我们玩个游戏。先不给示例，让它凭空输出。"

**命令（0-shot）**：
```
把 sample.log 中的每个错误提取出来，按这个格式输出：
- 错误级别
- 错误原因
- 建议操作
```

**预期输出**：
- AI 的输出格式大致对但细节不一致
- 每条输出的结构可能不同

**讲师话术**：
"大致对，但不统一。现在给它看两个例子。"

**命令（2-shot）**：
```
把 sample.log 中的每个错误提取出来。参考以下示例格式：

示例1：
输入：2026-05-24 10:23:15 ERROR DatabaseError: connection timeout after 30s
输出：{"级别":"ERROR", "原因":"连接超时", "建议":"检查数据库连接池配置"}

示例2：
输入：2026-05-24 10:23:45 ERROR InternalError: null value in column "user_id"
输出：{"级别":"ERROR", "原因":"空值违反非空约束", "建议":"在插入前校验 user_id 是否为空"}

现在分析 sample.log 中所有行。
```

**预期输出**：
- AI 严格遵循示例的 JSON 格式
- 每条输出的结构完全一致

**关键展示点**：
- 两个例子比写 100 字描述更有效
- Few-shot 的核心是"让 AI 看到你想要的形状"

**备用方案**：
- 如 2-shot 效果不明显，追加第 3 个示例

**衔出过渡**：
"现在 AI 的输出质量已经很好了，但这是面向人读的。如果你想把 AI 的输出接入自动化流程，需要的不是一段优美的散文，而是机器能解析的结果。"

---

### 步骤 4：结构化输出（JSON 约束 + Pydantic AI）

**前情过渡**：
"现在 AI 的输出质量已经很好了，但这是面向人读的。如果你想把 AI 的输出接入自动化流程，需要的不是一段优美的散文，而是机器能解析的结果。"

**讲师话术**（前半 — JSON 约束）：
"最简单的办法——在提示词里要求 JSON。"

**命令（JSON 约束）**：
```
分析 sample.log 中的所有日志行。
只输出纯 JSON 数组，不要任何解释文字。
格式：[{"time":时间, "level":级别, "message":消息, "category":分类}]
```

**预期输出**：
- 纯 JSON，可直接被程序解析

**讲师话术**（后半 — 引出问题）：
"看起来完美。但如果你把这个用在生产环境——每天跑几万次——AI 偶尔会输出格式不对的 JSON。多了一个逗号、少了一个引号，你的程序就崩了。更好的办法是让代码来定义数据结构，让库来保证类型安全。"

**演示 Pydantic AI 代码**（讲师在编辑器中打开 `scripts/pydantic_ai_example.py`，逐段讲解）：

```python
from pydantic import BaseModel
from pydantic_ai import Agent

# 1. 定义数据结构 —— 这就是你的"表单"
class LogEntry(BaseModel):
    time: str
    level: str
    message: str

class LogAnalysisResult(BaseModel):
    entries: list[LogEntry]
    total_errors: int
    summary: str

# 2. 创建 Agent，指定返回类型
agent = Agent("openai:gpt-4o", result_type=LogAnalysisResult)

# 3. 一句话调用，自动校验
result = agent.run_sync("分析 sample.log 中的所有日志行")

# result 是类型安全的对象，IDE 有自动补全
print(f"共 {result.total_errors} 条")
for entry in result.entries:
    print(f"[{entry.level}] {entry.message}")
```

**展示 Mermaid 原理图**（取自讲义）：

讲解要点：
- 第 1 步：你定义 Pydantic Model（这就是你想要的"表单"）
- 第 2 步：库自动把 Model 转成 JSON Schema 发给 LLM
- 第 3 步：LLM 按 Schema 返回结构化数据
- 第 4 步：Pydantic 校验——类型不对？字段缺失？自动把错误信息喂给 LLM 重试
- 结果：你拿到的永远是类型安全的 Python 对象

**关键展示点**：
- Prompt 约束 JSON = 你靠嘴说；Pydantic AI = 你靠代码保证
- 校验失败自动重试是核心亮点——人工处理格式错误太累了
- 适合生产环境：类型安全、IDE 补全、自动重试

**备用方案**：
- 若 Pydantic AI 安装演示不顺利，直接展示 Mermaid 图和代码片段，口头讲解原理
- 若 AI 仍然加了 Markdown 代码块标记，追加命令："不要用 ```json 包裹，直接输出 JSON"

**衔出过渡**：
"到目前为止，我们学的都是'一轮对话'的技巧——你写好提示词，AI 给你答案。但现实中复杂的开发问题不是一问一答能搞定的。接下来我们升级一下：怎么让 AI 自己先想清楚再回答，甚至自己去验证答案对不对。"
