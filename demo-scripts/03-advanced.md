# 演示脚本 03：提示词工程 · 进阶篇

> **运行环境**：Windows（CMD 或 PowerShell），命令已适配 Windows。Linux/macOS 用户将 `python` 换为 `python3`，`type` 换为 `cat`，`dir /s` 换为 `ls -R`。

## 场景准备

```bash
cd demo-project
pi
```

任务升级：不是简单地分析日志，而是要找出 parse.py 的 bug 并验证。

---

### 步骤 1：思维链（直接回答 vs 分步推理）

**前情过渡**：
"到目前为止，我们学的都是'一轮对话'的技巧——你写好提示词，AI 给你答案。但现实中复杂的开发问题不是一问一答能搞定的。接下来我们升级一下：怎么让 AI 自己先想清楚再回答。"

**讲师话术**：
"我们问同一个问题两遍——第一遍直接要答案，第二遍让它一步步想。"

**命令（直接回答）**：
```
parse.py 有什么 bug？
```

**预期输出**：
- AI 给出猜测性回答
- 可能正确，也可能跑偏
- 学员无法判断 AI 的推理是否可靠

**讲师话术**：
"答案看起来挺自信的——但它可能猜的。现在我们让它把脑子里的过程写出来。"

**命令（CoT）**：
```
请逐步分析 parse.py 是否有 bug。每一步：
1. 先看 sample.log 的内容，列出所有日志行及其级别
2. 再看 parse.py 的代码，理解它的解析逻辑
3. 手动模拟 parse.py 对 sample.log 的处理，预测输出
4. 对比预测输出和实际可能的正确输出
5. 如果有不一致，描述 bug

逐步写出你的分析过程，每一步给出结论后再进入下一步。
```

**预期输出**：
- AI 分步展示：
  - 步骤1：列出 6 行日志（4 ERROR + 1 WARN + 1 INFO）
  - 步骤2：正则 `r'... ERROR ...'` 只匹配含 ERROR 的行
  - 步骤3：预测输出 4 条
  - 步骤4：预期应输出 6 条
  - 步骤5：bug = 正则硬编码了 ERROR，漏掉 WARN/INFO

**关键展示点**：
- 每步推理可追踪，结论有据可查
- 如果结论错了，你能找到推理链上哪一步出了问题
- CoT 比直接回答更可靠，但回答更长、更贵——需要权衡

**备用方案**：
- 若 AI 仍然跳步，追加："请严格遵守 1→2→3→4→5 的顺序，每步标注'第N步：'"

**衔出过渡**：
"思维链是让 AI 在脑子里多转几圈。但真正的开发场景里，光想不够——AI 需要动手做事：运行命令、读文件、看报错，然后根据实际结果调整。这个'思考 + 行动 + 观察 + 再思考'的循环，就是 ReAct 模式——它也是后面要讲的 Agent 的核心机制。"

### 步骤 2：ReAct 模式（手动构造多轮对话）

**前情过渡**：
"思维链是让 AI 在脑子里多转几圈。但真正的开发场景里，光想不够——AI 需要动手做事。"

**讲师话术**：
"现在我们换一种方式跟 AI 交互——每轮我只让它做一件事，然后看结果，再决定下一步。这就是 ReAct 模式。"

**命令**（第一轮 — 思考+行动）：
```
我们一起来排查 parse.py 的问题。先运行 python parse.py sample.log，把输出和 sample.log 的原始内容对比。告诉我你发现了什么。
```

**预期输出**：
- AI 对比输出（4 条）和原始日志（6 行）
- 指出 WARN 和 INFO 行没有被解析

**命令**（第二轮 — 观察+再思考）：
```
现在读 parse.py 的源码，找出为什么 WARN 和 INFO 行被漏掉了。
```

**预期输出**：
- AI 指出第 9 行的正则 `ERROR` 只匹配一个单词
- 建议改为匹配任意日志级别

**命令**（第三轮 — 行动+验证）：
```
修改 parse.py 第 9 行，让正则可以匹配所有日志级别（ERROR、WARN、INFO 等），然后运行 python parse.py sample.log 验证输出变成 6 条。
```

**预期输出**：
- AI 执行修改
- 运行验证，输出 6 条

**关键展示点**：
- 每轮只做一件事：观察 → 思考 → 行动 → 再观察
- 这正是 Agent 的底层工作模式
- 但现在每一步都需要你手动发送指令

**备用方案**：
- 若 pi 不支持在交互中直接修改文件，改为让 AI 输出修改后的代码，讲师手动粘贴

**衔出过渡**：
"ReAct 模式好是好，但它会让对话越来越长。你每轮都带着前面所有的上下文——这就像跟同事聊了三个小时，开头说的重点他早忘了。AI 也一样，它有上下文窗口的限制。"

### 步骤 3：Agent 上下文组成与生命周期

**前情过渡**：
"ReAct 模式好是好，但它会让对话越来越长。"

**讲师话术**：
"你每轮对话都带着前面所有的上下文——AI 也一样，它有上下文窗口的限制。要理解这个限制怎么破，得先知道 Agent 的上下文到底由什么组成、每部分活多久。"

**演示 1 — 展示上下文三层结构图**（展示 Mermaid 图，取自讲义，逐一讲解三层）：

"一个 Agent 的上下文，每次发给 LLM 的 prompt，就是由这三块拼出来的。"

**系统提示（System Prompt）**：
- 包含：角色定义、行为约束、可用 Skill 列表
- 生命周期：整个 session 常驻，永远不会被压缩
- 占比：约 15-30%，固定不变
- "所以关键信息要放在这里——你要 Agent 始终记住的东西，写进 system prompt"

**工具定义（Tool Definitions）**：
- 包含：每个工具的名称、参数 schema
- 生命周期：整个 session 常驻
- 占比：约 5-10%，工具越多占越多
- "工具太多会吃掉对话空间——只给必要的"

**对话消息（Conversation Messages）**：
- 包含：用户消息、助手回复（含思考过程）、工具调用、工具结果
- 生命周期：不断累积，超限时触发压缩
- "这才是大头——对话越多，占的空间越大。突破窗口怎么办？压缩。"

**演示 2 — 展示上下文累积与压缩流程图**（展示 Mermaid 图，取自讲义，边指边讲）：

"看这条曲线——"

逐阶段讲解：
1. Session 刚开始：System + Tools 占 ~20%，还有 ~80% 给对话
2. 第 5 轮：上下文用了 ~50%
3. 第 15 轮：只剩 ~15%——危险！
4. 超过压缩阈值：触发 Compaction
5. 压缩后：旧消息→摘要（token 从几千→几百），加上最近 N 轮，回到安全水位 ~60%

**演示 3 — 压缩机制深入**：

"压缩不是简单删掉旧消息。它分三步走："

1. **选切割点**：保留最近 ~20K tokens，更早的进入压缩区
2. **生成摘要**：用一个轻量模型调用，把旧对话压缩成结构化摘要——目标、进度、决策、下一步
3. **迭代更新**：下次再压缩时，在已有摘要基础上更新，不是重新总结

"压缩后的摘要虽然不完美，但比'AI 忘了前面说过什么'要好得多。"

**演示 4 — System Prompt 保护实验**（如果时间允许）：

讲师示范：
1. 长对话中，随便写在对话中间的"记住：目标文件是 parse.py"——可能在第 15 轮被压缩掉
2. 但如果放在 System Prompt 里——永远不会丢

"这就是关键信息要放 System Prompt 的原因：它是上下文里唯一'永生'的部分。"

**关键展示点**：
- 上下文 = System Prompt（常驻）+ Tools（常驻）+ Messages（动态+可压缩）
- 压缩不是删消息，而是用 LLM 把旧对话总结成一段摘要
- 把关键约束放 System Prompt，把参考知识放 Skill 的 references，对话动态部分放心让压缩机制管理
- 压缩阈值和保留策略是可配置的——pi 默认预留 16K、保留最近 20K

**备用方案**：
- 若 Mermaid 图渲染不顺利，讲师在白板/纸上手画三层结构图
- 重点不是记住具体数字，而是理解"三层结构 + 压缩机制"这一对核心概念

**衔出过渡**：
"现在你的工具箱里已经有好几件利器了——CoT、ReAct、上下文管理。但每次用的时候都重新敲一遍提示词太傻了。程序员的本能反应是什么？封装复用。"

### 步骤 4：封装为 Skill — 提示词的"函数化"

**前情过渡**：
"现在你的工具箱里已经有好几件利器了——CoT、ReAct、上下文管理。但每次用的时候都重新敲一遍提示词太傻了。程序员的本能反应是什么？封装复用。"

**讲师话术**：
"你刚刚手动做的那个 CoT 分析流程——先看日志、再读源码、对比差异、定位根因——如果每次都要重新敲一遍提示词，太累了。我们把它封装成一个 Skill。"

**演示 1 — 展示 Skill 目录结构**（讲师在终端运行）：

```bash
dir /s skills\log-analyzer\          # Windows
# Linux/macOS 用: ls -R skills/log-analyzer/
```

输出：
```
skills/log-analyzer/
├── SKILL.md
├── scripts/
│   └── parse_log.py
└── references/
    └── log-patterns.md
```

**演示 2 — 打开 SKILL.md**（讲师在编辑器中打开 `skills/log-analyzer/SKILL.md`，逐段讲解）：

"看这个结构——YAML 头部是 Skill 的'身份证'，Agent 启动时扫描所有 Skill，只读这两行 metadata（~100 tokens），不会把整个文件都塞进上下文。"

```yaml
---
name: log-analyzer
description: 分析日志文件，找出错误根因并给出修复建议。
             当用户提到"日志分析"、"找 bug"时使用。
---
```

"下面的 Markdown 正文是真正的'函数体'——Agent 只在需要时才加载。6 步分析流程，本质上就是把刚才的 CoT 提示词封装成了结构化的指令模板。"

**演示 3 — 展示渐进式加载原理**（展示 Mermaid 图，取自讲义）：

讲解要点：
- Agent 启动 → 扫描 skills/ 目录 → 只加载 metadata（~100 tokens）
- 用户说"帮我分析日志" → 匹配到 log-analyzer skill → 加载完整 SKILL.md
- 需要查日志模式 → 按需加载 references/log-patterns.md
- 三层加载：metadata（常驻）→ 指令（按需）→ 参考（进一步按需）
- 这就是 agentskill.io 标准的核心设计：Progressive Disclosure

**演示 4 — 展示辅助工具**（讲师打开 `skills/log-analyzer/scripts/parse_log.py`）：

"Skill 不只是文本模板，还可以带工具脚本。这个 `parse_log.py` 是 log-analyzer skill 的辅助脚本，Agent 在执行 Skill 时会直接调用它来快速统计日志文件。"

**关键展示点**：
- Skill 是提示词模板的"工业化"版本——不只是 .txt 文本，而是带 metadata、指令、脚本、参考文档的完整包
- Agent Skills 是开放标准（agentskill.io），一份 Skill 可以在 Claude Code、Gemini CLI、Cursor、GitHub Copilot 等 40+ 平台通用
- Progressive Disclosure（渐进式加载）让 Agent 能管理大量 Skill 而不爆上下文
- 对比"保存为 .txt 文件"vs"写成符合规范的 Skill"：后者让 Agent 自动发现、自动匹配、按需加载

**备用方案**：
- 若编辑器展示不顺利，直接在终端用 `type skills\log-analyzer\SKILL.md`（Windows）展示（Linux/macOS：`cat skills/log-analyzer/SKILL.md`）
- 重点不在于看每一行代码，而在于理解"封装复用"和"渐进式加载"这两个概念

**衔出过渡**：
"回顾一下我们现在做的事：写好提示词 → 看 AI 输出 → 判断结果行不行 → 不行就改提示词再来 → 再看输出……每一步都是你手动操作。有没有一种东西，能自动完成这个循环？"
