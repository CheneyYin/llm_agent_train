# 演示脚本 03.5：Tool 调用原理与实践

## 场景准备

```bash
cd demo-project
pi
```

> **运行环境**：Windows（CMD 或 PowerShell），命令已适配 Windows。Linux/macOS 用户将 `python` 换为 `python3`。

---

### 步骤 1：Tool 调用的本质

**前情过渡**：
"ReAct 模式里有一个关键步骤——'行动'。AI 自己不会动手——你需要给它工具。"

**讲师话术**：
"AI 怎么'行动'？说到底很简单——LLM 只是在响应中输出一段 JSON。看这个例子。"

**展示**（讲师指向 Mermaid 图）：

讲解要点：
- LLM 输出 `{"name":"read_file","arguments":{"path":"sample.log"}}` — 仅此而已，没有一个字是"执行"
- Agent Runtime 拦截这个 JSON → 真正调用 `read_file("sample.log")` → 拿到文件内容
- 把文件内容作为新消息回传给 LLM → LLM 继续推理
- "LLM 是大脑，Tool 是手。大脑说'读那个文件'，手去读，触觉反馈回大脑。"

**关键展示点**：
- LLM 不知道文件系统长什么样——它只输出 JSON
- 执行工具的是 Runtime，不是 LLM
- 工具结果作为对话消息追加——LLM "看到"了执行结果

**衔出过渡**：
"这个流程的每一步都有具体的钩子。我们对照 pi 的源码，看一个 Tool 调用从出生到死亡经历了什么。"

### 步骤 2：Tool 调用协议 6 步

**前情过渡**：
"这个流程的每一步都有具体的钩子。"

**讲师话术**（指向 Mermaid 时序图，逐步讲解）：

| 步骤 | 讲师话术 |
|------|---------|
| ① LLM 输出 | "第一步，LLM 流式输出 tool call。注意——它先输出名字，再输出参数。参数可能很长，是分片段（delta）传过来的。" |
| ② 校验 | "第二步，Runtime 做两件事。一是 TypeBox schema 验证——参数类型对不对、必填字段有没有。二是 `beforeToolCall` 钩子——可以做权限控制，返回 `{block: true}` 就能拦截。" |
| ③ 执行 | "第三步，真正执行。`tool.execute()` 接收参数、signal（用于取消）、onUpdate（用于推送进度）。" |
| ④ 结果 | "第四步，工具返回结构化结果。注意 `terminate` 字段——设为 true 可以让 Agent 停止循环。" |
| ⑤ 后处理 | "第五步，`afterToolCall` 钩子。可以重写结果、标记错误。比如工具返回了完整日志，这里可以 trim 只保留关键部分。" |
| ⑥ 追加 | "第六步，toolResult 消息追加到对话。下轮 LLM 调用时，这条消息就在上下文里了——LLM 据此决定下一步。" |

**pi 源码对照表**（讲师在幻灯片上指认）：

"六步在 pi 里对应六个钩子。你可以看到——这个协议是完整的、工业级的。"

**关键展示点**：
- 流式更新的意义：工具运行时可以实时推送进度，不用等待完成
- 权限门禁：`beforeToolCall` 是安全边界
- 并行执行：多个 tool call 可以并发

**衔出过渡**：
"原理懂了，我们来写一个真正的工具——用 pi 的 TypeScript 扩展机制，给 Agent 的手上加一根手指。"

### 步骤 3：实战 — 注册自定义 Tool

**前情过渡**：
"原理懂了，我们来写一个真正的工具。"

**讲师话术**（打开 `scripts/log-tools.ts`，逐段讲解）：

"一个 Tool 有三个核心要素。我们先看第一个——Schema。"

**第 1 段 — Schema**：
```typescript
parameters: Type.Object({
  filepath: Type.String({ description: "日志文件路径" }),
}),
```
"这个 Schema 做了两件事：① 告诉 LLM '这个工具需要一个 filepath 参数' ② 在运行时校验——如果 LLM 传了数字，TypeBox 直接拒绝。"

**第 2 段 — Execute**：
```typescript
execute: async (toolCallId, params, signal, onUpdate) => {
  onUpdate({ content: [{ type: "text", text: "正在读取..." }], details: {} });
  // ...业务逻辑...
  return { content: [...], details: {...} };
},
```
"Execute 是真正的业务逻辑。注意 onUpdate——可以在执行中推送进度，用户能看到'正在读取文件...'。"

**第 3 段 — Hook**：
```typescript
pi.on("tool_call", async (event) => {
  if (!path.endsWith(".log")) {
    return { block: true, reason: "只允许分析 .log 文件" };
  }
});
```
"Hook 做权限控制。这个例子只允许读取 .log 文件——如果你让 Agent 读 /etc/passwd，它会拒绝。"

**效果演示**（在 pi 中输入）：
```
用 count_log_levels 工具分析 sample.log
```

预期输出：Agent 调用 `count_log_levels`，返回 `{"ERROR": 4, "WARN": 1, "INFO": 1}`。

**三要素总结**（讲师指向幻灯片上的表格）：

"记住这三个要素——Schema 定义接口、Execute 执行逻辑、Hook 控制权限。任何 Agent 框架的工具系统，本质上都是这三件事。"

**备用方案**：
- 若 pi extension 加载不顺利，直接展示代码文件 + Mermaid 图 + 口头讲解，不依赖实时执行
- 预先准备一张截图展示 Agent 调用 count_log_levels 的输出
