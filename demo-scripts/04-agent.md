### 步骤 1：什么是 Agent — 概念演示

**前情过渡**：
"回顾一下我们现在做的事：写好提示词 → 看 AI 输出 → 判断结果行不行 → 不行就改提示词再来 → 再看输出……每一步都是你手动操作。有没有一种东西，能自动完成这个循环？它自己思考、自己调用工具、自己看结果、自己决定下一步？这个东西就叫 Agent。"

**讲师话术**：
"Agent 的本质就一句话：把'思考→行动→观察→再思考'的循环自动化。"

展示 Mermaid 图（取自讲义）：

"左边的图是我们刚才的 ReAct 演示——每一步都要你手动发指令。右边的图是 Agent——它自己跑整个循环。代码是一样的代码，工具是一样的工具，区别只是谁在按'下一步'按钮。"

**关键展示点**：
- Agent 不是新技术，它就是 ReAct 的自动化版本
- 你从"操作者"变成"监督者"

**衔出过渡**：
"Agent 听起来神秘，但拆开看核心就是一个简单的四步循环——和我们刚才手动做的 ReAct 几乎一模一样，只是现在 AI 自己来驱动这个循环了。"

### 步骤 2：Agent 核心循环拆解

**前情过渡**：
"Agent 听起来神秘，但拆开看核心就是一个简单的四步循环。"

**讲师话术**：
"我们看一下 pi agent 在执行一个任务时后台发生了什么。下面是它处理'分析 sample.log 并修复 parse.py'这个任务时的执行日志。"

**回放日志关键片段**（讲师逐段讲解）：

```
[Agent 收到任务] "分析 parse.py 和 sample.log，找出 bug 并修复"

--- 第1轮 ---
[思考] 我需要先了解数据。sample.log 是日志文件，先看看内容。
[行动] read_file("sample.log")
[观察] 6行日志：4×ERROR, 1×WARN, 1×INFO

--- 第2轮 ---
[思考] 日志有 WARN 和 INFO 级别。我需要检查 parse.py 是否正确处理所有级别。
[行动] read_file("parse.py")
[观察] parse_log 函数第9行正则：r'... ERROR ...' ——只匹配 ERROR

--- 第3轮 ---
[思考] 找到 bug：正则在 ERROR 处硬编码，漏掉 WARN/INFO。修复方案：用 \w+ 匹配任意级别。
[行动] edit("parse.py", line=9, old="ERROR", new="\w+")  # 实际更复杂
[观察] 修改完成

--- 第4轮 ---
[思考] 需要验证修复效果
[行动] run("python parse.py sample.log")
[观察] 输出 6 条记录 ✓ 修复成功

--- [循环结束] ---
[Agent 报告] "parse.py 的 bug 已修复。根因：第9行正则只匹配 ERROR 导致 WARN/INFO 被忽略。
修复：将 ERROR 改为 \w+ 匹配所有日志级别。验证：运行 parse.py sample.log 输出 6 条，与日志行数一致。"
```

**讲师在每轮停顿，指出现场对应的步骤**：

| 轮次 | 步骤 | 讲师点评 |
|------|------|---------|
| 第1轮 | 理解→行动→观察 | "它自己决定先看数据，不是我们告诉它的" |
| 第2轮 | 理解→行动→观察 | "它对比了日志和代码，发现不匹配——就像你做 code review" |
| 第3轮 | 理解→行动→观察 | "找到 bug，自动修复——注意它没有问'要不要修'，因为它理解任务目标是修复" |
| 第4轮 | 理解→行动→观察 | "自动验证——程序员的好习惯，Agent 也有" |

**关键展示点**：
- 每轮都是"理解→行动→观察"的循环
- Agent 自己决定每一步用什么工具、读什么文件
- 不需要你写 CoT 提示词——Agent 内部就是这么工作的

**衔出过渡**：
"刚才拆开的四步循环里有一个关键能力——'选择工具'。但 Agent 不只有工具，它还需要知道'怎么做这件事'。这就是 Skill——它不是手，是知识库。"

### 步骤 3：Skill 规范与原理

**前情过渡**：
"刚才拆开的四步循环里有一个关键能力——'选择工具'。但 Agent 不只有工具。"

**第1节 — Skill vs Tool 对比**（讲师指向幻灯片上的对比表和 Mermaid 图）：

"Tool 是手——执行操作，结果追加到对话消息里。Skill 是知识库——Agent 启动时就注入到 System Prompt，整个 session 都记得。Tool 按需调用，用完就忘；Skill 常驻内存，随时可用。"

**第2节 — 渐进式加载**（讲师打开 `skills/log-analyzer/SKILL.md`，指向幻灯片上的 Mermaid 图）：

"看这个目录结构——SKILL.md 的核心是 YAML 头部加 Markdown 正文。Agent 启动时只读 YAML 头部的 name 和 description，才 ~100 tokens。只有用户任务匹配到 description，才加载完整的 SKILL.md 正文。这还没完——references/ 里的参考文档是第三层，执行过程中才按需加载。"

"这个设计叫渐进式加载（Progressive Disclosure）。靠这三层，Agent 可以管理几十上百个 Skill 而不会撑爆上下文。"

**第3节 — 演示 Skill 生命周期**（讲师在 pi 中操作）：

```bash
# 1. pi 启动时自动扫描 skills/ 目录
pi

# 2. 查看 available_skills，确认 log-analyzer 已加载
# 3. 输入任务——Agent 自动匹配 Skill
"帮我分析 sample.log 的错误根因"
```

预期效果：
- Agent 自动匹配到 `log-analyzer` Skill（description 匹配）
- 按 Skill 定义的 6 步流程执行（先读日志 → 再读源码 → 对比 → 定位 → 修复建议 → 验证）
- 需要时按需加载 `references/log-patterns.md`

**关键展示点**：
- Skill 和 Tool 的区别：知识库 vs 执行器
- 渐进式加载三层：metadata（常驻）→ 指令（按需）→ 资源（进一步按需）
- agentskill.io 是开放标准，40+ 平台通用

**备用方案**：
- 若 pi 启动不顺利，直接展示 `skills/log-analyzer/` 目录结构和 SKILL.md，口头讲解渐进式加载

**衔出过渡**：
"Skill 让 Agent 能复用知识和流程，但不是所有场景都适合写成 Skill。接下来我们聊聊 Agent 的边界在哪——什么该交给 Skill，什么该自己盯着。"

### 步骤 4：Agent 的局限与边界

**前情过渡**：
"Skill 让 Agent 能复用知识，但不是所有场景都适合写成 Skill。"

**讲师话术**：
"Agent 有三大边界，每个使用 Agent 的开发者都应该知道。"

**边界一：任务越模糊，Agent 越容易跑偏**

"刚才我们的任务是明确的：'分析 parse.py 找 bug 并修复'。如果你说'帮我改进这个项目'——Agent 不知道该做什么，可能乱改代码。"

**边界二：Agent 的修改需要人工 review**

"Agent 的代码和人类一样可能有 bug。它修好了 parse.py 的正则，但也许它的修复引入了新问题。永远 review Agent 的改动，就像 review 同事的 PR。"

**边界三：在隔离环境中运行 Agent**

"Agent 有修改文件的能力。在生产服务器上直接跑 Agent？不建议。用 git worktree、Docker 容器或 VM 来隔离 Agent 的操作环境。"

展示 Mermaid 决策树（取自讲义）：

"记住这个决策流程：任务明确 + 范围可控 + 结果可验证 → 用 Agent；否则 → 自己动手或监督执行。"

**关键展示点**：
- 不是所有任务都适合 Agent
- 三个条件：目标明确、范围受控、可验证
- Agent 是工具，不是替代品

**衔出过渡**：
"我们从'怎么写好一句话'开始，一路走到了'怎么让 AI 自己完成整个任务'。让我们退一步，回顾一下这一小时我们学到了什么，更重要的是——明天上班你能立刻用上什么。"
