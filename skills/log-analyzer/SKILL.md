# 日志分析器

## 触发条件

当用户需要：
- 分析应用错误日志
- 从日志中找出 bug 根因
- 排查线上问题
- 对比日志输出和源代码逻辑

## 分析流程

### 第1步：理解数据

首先读取日志文件，统计：

- 总行数
- 按日志级别（ERROR / WARN / INFO）分类数量
- 错误类型分布（DatabaseError、InternalError 等）
- 时间分布（是否有集中爆发时段）

参考 `references/log-patterns.md` 中的常见日志模式辅助判断。

### 第2步：阅读源码

如果用户提供了源代码文件，读取并理解其逻辑。重点关注：

- 日志解析 / 处理逻辑
- 正则表达式和字符串匹配
- 异常处理机制
- 边界条件检查

### 第3步：对比分析

将日志的**实际内容**和代码的**预期行为**对比：

- 代码预期处理哪些级别的日志？
- 实际日志中包含哪些级别的日志？
- 是否存在不匹配？（如代码只处理 ERROR，但日志含 WARN/INFO）

### 第4步：定位根因

找出导致不匹配的具体代码位置和原因。输出格式：

```
- 文件：<文件路径>
- 行号：<行号>
- 问题代码：<代码片段>
- 根因：<为什么会导致当前问题>
```

### 第5步：给出修复建议

对每个发现的 bug，给出：

- 修复方案（具体的代码修改）
- 修改后的代码片段
- 修复后如何验证（运行什么命令、期望什么输出）

### 第6步：验证修复

如果用户允许，执行以下验证：

1. 运行修改后的程序
2. 检查输出是否符合预期
3. 确认没有引入新的问题

## 辅助工具

运行 `scripts/parse_log.py <日志文件>` 可以快速统计日志文件的基本信息（行数、级别分布）。

## 输出格式

分析完成后，输出以下结构化报告：

```
## 日志分析报告

### 基本信息
- 日志文件：<路径>
- 总行数：<N>
- 级别分布：ERROR=X, WARN=Y, INFO=Z

### 发现问题
1. <问题1描述>
   - 位置：<文件:行号>
   - 根因：<描述>
   - 修复：<方案>

### 建议
<总结性建议>
```
