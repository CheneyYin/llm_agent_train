第二部分：提示词工程 · 基础篇

15 分钟

输入的质量决定输出的质量

什么是提示词工程？

你对 AI 说"帮我改代码"→ 它猜。
你说"分析 sample.log 的 ERROR 行，按类型归类"→ 直接出结果。
输入质量决定输出质量。

提示词工程 = 学会怎么跟 AI 说话

技巧一：角色设定

告诉 AI 你是谁，它才知道怎么回答 — 就像点菜前告诉厨师口味。
回合提示词效果
无角色"分析 sample.log"泛泛回答
有角色"你是资深 Python 后端开发，负责日志分析。请分析 sample.log"专业分析视角

技巧二：明确指令与约束

回合提示词
模糊"帮我分析这个日志文件"
精确"分析 sample.log：① 按错误类型分组 ② 统计次数 ③ 按次数降序排列"

技巧三：Few-shot 示例

给一两个输入→输出的例子，AI 照猫画虎。

回合提示词效果
0-shot"把错误信息提取出来"格式不确定
2-shot"示例1：ERROR timeout → {级别:ERROR, 原因:timeout}。示例2：ERROR null → {级别:ERROR, 原因:null_value}。现在分析所有行。"严格遵循格式

技巧四：结构化输出

Prompt 约束 JSON

"只输出 JSON，不要其他文字"
[{level, message, count}]

⚠ 偶有格式错误，依赖 LLM 输出质量

Pydantic AI — 代码定义 Schema

class LogEntry(BaseModel):
    time: str
    level: str
    message: str

agent = Agent(
    "openai:gpt-4o",
    result_type=LogAnalysisResult
)

✓ 自动校验 + 重试，类型安全

让 AI 学会先思考再回答

技巧五：思维链（CoT）

回合提示词
直接"parse.py 有什么 bug？"
CoT"逐步分析：① 看 sample.log 内容 ② 读 parse.py 逻辑 ③ 对比预期和实际 ④ 描述 bug"

CoT 效果对比

方式优点缺点
直接回答快速、tokens 少可能猜错，无法追溯推理
思维链推理可追踪，结论有据tokens 多，回答长

技巧六：ReAct 模式

ReAct = 思考→行动→观察→再思考。所有 Agent 的底层工作模式。

flowchart LR T[思考 Thought] --> A[行动 Action] A --> O[观察 Observation] O --> T

ReAct 演示

第1轮：思考"先运行 parse.py"→ 行动：执行 → 观察：只输出4条，日志有6行

第2轮：思考"漏掉了 WARN/INFO"→ 行动：读取源码 → 观察：正则只匹配 ERROR

第3轮：思考"改正则"→ 行动：修改代码 → 观察：修复完成

第4轮：思考"验证"→ 行动：运行 → 观察：输出6条 ✓

这就是 Agent 的底层工作模式

System Prompt 是系统进程，永不清理。

压缩机制与核心原则

压缩三步：① 选切割点（保留最近 ~20K tokens）→ ② 生成结构化摘要 → ③ 迭代更新已有摘要

原则说明
关键信息放 System Prompt永远不被压缩，最安全
长对话信任摘要虽不完美，但比"AI 忘了"好得多
工具输出精简别把 10MB 日志全塞进上下文

工具专题：Tool 调用原理与实践

8 分钟

LLM = 大脑 · Tool = 手 · Runtime = 神经系统

Tool 调用的本质

LLM 只输出 JSON — 不执行任何操作

{"name":"read_file","arguments":{"path":"sample.log"}}
sequenceDiagram participant L as LLM participant R as Agent Runtime participant T as 真实工具 L->>R: 输出 tool call JSON R->>T: read_file("sample.log") T-->>R: 文件内容... R->>L: 追加 tool result 消息 L->>L: 根据结果继续推理

LLM = 大脑 · Tool = 手 · 结果 = 触觉反馈

Tool 调用协议 — 6 步生命周期

sequenceDiagram participant L as LLM participant R as Agent Runtime participant T as Tool L->>R: ① 输出 tool call R->>R: ② Schema 校验 + 权限检查 R->>T: ③ execute(params) T-->>R: ④ AgentToolResult R->>R: ⑤ afterToolCall 后处理 R->>L: ⑥ 追加 toolResult 到对话

pi 源码对照

步骤pi 钩子作用
① LLM 输出message_update流式接收 tool call 参数
② 校验validateToolArguments() + beforeToolCallTypeBox schema + 权限门禁
③ 执行tool.execute(id, params, signal, onUpdate)实际调用，支持流式进度
④ 结果AgentToolResult结构化结果 + terminate 信号
⑤ 后处理afterToolCall重写结果、标记错误
⑥ 追加context.messages.push()LLM 下轮带上工具结果

实战：注册自定义 Tool

场景：给 Agent 装一个 count_log_levels 工具

// extensions/log-tools.ts
export default function (pi: ExtensionAPI) {
  // ① Schema — 告诉 LLM 工具的参数格式
  pi.registerTool({
    name: "count_log_levels",
    label: "统计日志级别",
    parameters: Type.Object({
      filepath: Type.String({ description: "日志文件路径" }),
    }),
    // ② Execute — 实际业务逻辑
    execute: async (id, params, signal, onUpdate) => {
      onUpdate({ content: [{ type: "text", text: "读取中..." }], details: {} });
      // ...统计日志级别...
      return { content: [{ type: "text", text: JSON.stringify(counts) }], details: {} };
    },
  });
  // ③ Hook — 权限控制
  pi.on("tool_call", async (event) => {
    if (!event.filepath.endsWith(".log"))
      return { block: true, reason: "只允许 .log 文件" };
  });
}

什么是 Agent？

自驾 vs 打车
提示词工程 = 你开车，方向盘油门刹车全自己操作
Agent = 你打车，告诉司机目的地，可以看风景回消息——但得确认司机没走错路

手动交互

flowchart TD U1[你写提示词] --> A1[AI 回复] A1 --> U2[你看结果] U2 --> U3[你判断下一步] U3 --> U4[你再写提示词] U4 --> A2[AI 再回复] A2 --> U5[...]

Agent 自主循环

flowchart TD T1[理解任务] --> T2[选择工具] T2 --> T3[执行工具] T3 --> T4[观察结果] T4 --> T5{完成?} T5 -->|否| T1 T5 -->|是| T6[返回结果]

Agent 的局限与边界

大路司机认识，小路岔口需你指路。通用任务 Agent 很稳，特定细节仍需你介入。

flowchart TD Q1{任务目标明确?} -->|是| Q2{操作范围受控?} Q1 -->|否| A1[手动做] Q2 -->|是| Q3{结果可自动验证?} Q2 -->|否| A2[监督执行：每步确认] Q3 -->|是| A3[信任 Agent：自动完成] Q3 -->|否| A2

三个安全原则

  1. 不盲信：Agent 输出和人类代码一样需要 review
  2. 隔离环境：用 git worktree / Docker 限制操作范围
  3. 可回滚：所有修改都应可撤销（git commit 是最好保险）
